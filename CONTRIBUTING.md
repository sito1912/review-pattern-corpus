# コントリビューションガイド

`review-pattern-corpus` へのコントリビューションでは、レビューコーパス収集とパタンランゲージ保守の中核を、特定のAIエージェントやモデルから独立させることを重視します。

## 開発方針

- CLIは、Codex、Claude Code、その他の特定エージェントに結合しない。
- MVPではAIエージェントを直接実行しない。
- 依存は最小限にし、実用上問題がなければGo標準ライブラリを優先する。
- GitHub API利用は保守的にし、レート制限とタイムアウトを尊重する。
- 時刻の保存と期間比較にはUTCを使う。
- 生のレビューコーパスデータはデフォルトでコミットしない。

## セットアップ

このリポジトリはGo moduleです。

```sh
go test ./...
```

ローカルでGitHub APIを使う確認をする場合は、`GITHUB_TOKEN` または `GH_TOKEN` を設定するか、GitHub CLIで認証してください。

```sh
gh auth login
```

## ドキュメント更新

挙動や形式を変更するPull Requestでは、関連ドキュメントも更新してください。

- プロダクト挙動やスコープの変更: `docs/product-requirements.md`
- JSONL、catalog、パタン形式の変更: `docs/data-and-pattern-formats.md`
- マイルストーンやロードマップの変更: `docs/milestones.md`
- ユーザー向け概要や導入導線の変更: `README.md`、`docs/install.md`
- セキュリティ、プライバシー、保存方針の変更: `docs/security-and-privacy.md`

## テスト

コード変更では、少なくとも次を実行してください。

```sh
go test ./...
```

GitHub APIやCLIの入出力を変える場合は、ローカルテストに加えて、利用者が観測するエラーメッセージ、標準出力、標準エラー、出力ファイル名も確認してください。

## Pull Requestの観点

Pull Requestでは次を説明してください。

- 何を変えたか。
- なぜ必要か。
- MVPスコープを広げていないか。
- 生成されるJSONL、プロンプト、パタンファイルに互換性の影響があるか。
- 実行したテスト。

レビューでは次を優先して確認します。

- 特定AIエージェントへの結合が入っていないか。
- 生のレビューコーパスデータが意図せずコミットされていないか。
- UTC対象期間の境界が正しいか。
- botや生成コメントがコーパスへ混入しないか。
- GitHub APIに不要な負荷を生まないか。
- パタンファイルが抽象ガイダンスではなく、生コメントや生コードの保存になっていないか。
